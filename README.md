# Travel Mini Macau · 澳门出行助手

> 快速实现Agent应用与语言用户界面（LUI）的澳门旅游智能对话助手

---

## 1. 项目简介

### 1.1 项目背景

本项目通过调用大语言模型与地图工具，搭建旅游推荐内容的 Agent 助手。同时结合 LUI 控制地图界面，当对话触发地图查询相关意图时，左侧地图界面会同步展现推荐的地点或导航路径。

原则上，本项目可以推理任意地点（例如北京），但由于内置的静态地图数据主要围绕**澳门区域**，因此在澳门的信息呈现与地理空间推理上会更加准确。

### 1.2 项目目的

- **学习分享**：目前 Agent Loop 架构已经非常成熟，同时已经迈向 Agentic RL 的技术境界，本项目希望在当前 AI Coding 流行的氛围下，探索人机协作能将一个简单的项目推进到什么地步。
- **工作复现**：结合近期在智能座舱领域的智能体开发工作，希望将地理空间推理相关任务快速复现和留存学习记录，用于讨论未来大模型和智能体的技术课题。

## 2. 使用方法

1. 打开网页后，点击右上角 **设置（⚙️）** 图标。
2. 填入大模型配置：
   - `base_url`：OpenAI 兼容格式的 API 地址
   - `api_key`：你的 API Key
   - `model_type`：模型名称（如 `Qwen/Qwen3.5-27B`）
3. 填入高德地图 API Key（用于地理编码、POI 搜索、路径规划等 MCP 工具能力）。
   - 详见 [高德地图 MCP 使用方法](https://lbs.amap.com/api/mcp-server)
4. 返回对话界面，输入问题即可开始对话。对话过程中若触发地图相关意图，左侧地图将同步进行信息呈现（高亮景点、显示路线等）。

> **本地运行**：由于浏览器安全策略限制，请通过本地 HTTP 服务器打开（如 `python3 -m http.server 8080`），然后访问 `http://localhost:8080/index.html`。
>
> **Github pages**：在git仓设置对应的 Github pages 设置。

## 3. 项目底层实现方法

1. **Agent 后端**：使用 [Qwen-Agent](https://github.com/QwenLM/Qwen-Agent) 搭建 Agent 核心链路。选择该框架是因为其结构纯净、内置抽象类工具少，便于快速二次开发。
2. **地图工具**：结合高德地图 MCP 工具，实现地理编码、POI 搜索等地图相关的功能。
3. **Skills 增强**：内置文本化的技能描述（Skills.md），帮助 Agent 在进行工具发现时获得参考上下文，提升意图匹配工具的准确率。
4. **前端界面**：本项目先对Agent框架进行后端开发，前端界面由 AI Coding 辅助生成纯静态网页。

## 5. 未来方向

1. **地图查询与本地公交数据融合**
   目前地图界面内置了比较多静态的澳门地理数据和澳门公交 API (目前已失效)，但澳门相关数据的格式与高德 API 工具的结果存在较多不对称（例如在路线经纬度截取与数据对齐上实现难度较大），这个交互功能是未来开发需求。

2. **全双工语音交互**
   现有模型采用纯文本交互。若增加 ASR → NLU → TTS 级联系统，在已有较长的对话链路下会导致端到端时延显著增加。基于人机交互高响应的目标，需要在**回答价值策略**、**全双工/半双工 trade-off** 等方向上进行设计，以此来保证交互流畅度。

3. **模型地理空间推理能力**
   在规格较小的模型下，生成 UI 动作指令时往往会出现格式错误。文本/多模态模型能否准确推理地理空间位置、坐标映射与路径规划，仍然是一个极具研究价值的课题。

4. **AI 代码协作属于人机交互领域的课题**
   本项目主观判断AI coding 能够让开发者快速实现一个完整项目。因此，AI coding 在人机交互领域本身也属于一个值得探索的课题——即**如何让 AI 更好地增强人机交互体验**（这与全双工语音交互的方向一致）。
